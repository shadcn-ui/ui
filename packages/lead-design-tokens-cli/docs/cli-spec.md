## 0. Why this document exists

The CLI is the keystone of the Lead design system. Every decision in `DESIGN-DECISIONS.md` either becomes CLI behavior or becomes irrelevant — there's no third option, because `DESIGN.md` and all developer-consumable artifacts are CLI output. This document is the contract between the rationale and the implementation. Every non-trivial CLI behavior cites the decision it implements using `[D##]` references back to the decisions log in §13 of `DESIGN-DECISIONS.md`.

If the CLI ever does something this document doesn't describe, either the CLI is wrong or this document is stale. No third option there either.

## 1. Pipeline overview

```
┌─────────────────────┐     ┌──────────────────────┐
│  Paper Source       │     │ /tokens/authored/    │
│  tokens.paper.json  │     │  approvedPairs.json  │
│  sections/*.md      │     │  sourceExceptions    │
└──────────┬──────────┘     └──────────┬───────────┘
           │                           │
           ▼                           │
   ┌───────────────┐                   │
   │    import     │                   │
   └───────┬───────┘                   │
           ▼                           │
   /tokens/raw/paper/                  │
           │                           │
           ▼                           │
   ┌───────────────┐                   │
   │   normalize   │                   │
   └───────┬───────┘                   │
           ▼                           │
   /tokens/normalized/*.json           │
           │                           │
           └──────────┬────────────────┘
                      ▼
              ┌───────────────┐
              │     lint      │     reads normalized + authored
              └───────┬───────┘
                      ▼
              ┌───────────────┐
              │     build     │
              └───────┬───────┘
                      ▼
              /generated/*, DESIGN.md
```

Four tiers of token files. Ownership is strict:
- `/tokens/raw/paper/` — imported by CLI, not edited afterward.
- `/tokens/authored/` — hand-edited by designers / design-system owner, committed directly. Schema-validated.
- `/tokens/normalized/` — generated by CLI `normalize`. No direct edits.
- `/generated/` — generated by CLI `build`. No direct edits.

Each stage has a single responsibility and hands off clean, predictable output to the next. Stages are re-runnable and idempotent; running `normalize` twice on the same raw input produces byte-identical normalized output.

## 1.5 Shared pre-stage: `check-exceptions`

### 1.5.1 Purpose

Parse and validate `/tokens/authored/sourceExceptions.json` once per pipeline run, producing a resolved exception set that both `normalize` and `lint` consume. Exists because L7 tiebreaks influence **what normalize emits** (which variant's value becomes the canonical role value), not just what lint reports. Two stages can't each parse exceptions independently without risking divergent interpretations.

### 1.5.2 Invocation

```
design-tokens check-exceptions [--json]
```

Also runs implicitly as a pre-step of both `normalize` and `lint`. If it fails, neither proceeds.

### 1.5.3 Behavior

1. Expected state: `/tokens/authored/sourceExceptions.json` exists (seeded by `import`, §2.5). If it's missing, emit an empty exception set and a warning — this path is defense-in-depth for checkouts that somehow skipped `import`, not the expected first-run behavior. `[D37]`
2. Validate file against `sourceExceptions.schema.json`. Schema enforces:
   - `rule` enum: `"L5-size-floor" | "L7-weight-mismatch" | "L7-tracking-mismatch"`.
   - `tiebreak` enum: `"leading" | "no-leading"`. Required for L7 rules, forbidden for L5.
   - `resolvedBy` format: ISO date `YYYY-MM-DD` or quarter shorthand `YYYY-QN` (quarter 1–4 only).
   - `tokenPath` must be a dotted string; existence in normalized output is checked separately.
3. Resolve each `resolvedBy` to an absolute instant in `America/New_York`:
   - ISO `YYYY-MM-DD` → end of day (23:59:59) in America/New_York.
   - `YYYY-Q1` → March 31, 23:59:59 America/New_York.
   - `YYYY-Q2` → June 30, 23:59:59 America/New_York.
   - `YYYY-Q3` → September 30, 23:59:59 America/New_York.
   - `YYYY-Q4` → December 31, 23:59:59 America/New_York.
4. Check for duplicate `(tokenPath, rule)` pairs. Fail on duplicate — each token/rule combination gets at most one exception.
5. Tag each exception with `overdue: true` if its resolved instant is earlier than `now()` in America/New_York. Non-blocking; informational for downstream lint output.
6. Emit the resolved exception set to stdout (or in-memory to callers) as JSON, including the resolved instants for transparency.

### 1.5.4 Failure modes

- Schema violation (missing required field, invalid enum value, bad date format).
- Duplicate `(tokenPath, rule)` pair.
- L7 exception without `tiebreak`.
- L5 exception with `tiebreak`.

Overdue `resolvedBy` dates do NOT fail; they tag `overdue: true` and surface in lint output.

## 2. Command: `design-tokens import`

### 2.1 Purpose

Bring the paper-authored token library into the repo's raw snapshot without semantic modification. This is a **validate, copy, and manifest** step. Separating it from `normalize` means we can always diff what was authored on paper against what the CLI restructured.

### 2.2 Signature

```
design-tokens import [--source <path>] [--force]
```

- `--source` — directory containing the paper source bundle. Defaults to `tokens/source/paper`.
- `--force` — overwrite an existing raw snapshot.

### 2.3 Expected inputs

One canonical structured source file plus human-readable sections:

- `tokens.paper.json` — machine-readable paper source, validated against `schemas/paperTokens.schema.json`.
- `sections/*.md` — human-readable library sections. Copied into raw for provenance, but not parsed as token truth.

### 2.4 Outputs

```
/tokens/raw/paper/
  tokens.paper.json
  sections/*.md
  _manifest.json           ← source paths, import timestamp, file hashes
```

The raw snapshot remains a faithful copy of the paper source. Naming transformations happen in `normalize`, not `import`.

### 2.5 Step-by-step behavior

1. Verify `--source` directory exists and contains `tokens.paper.json`. If missing, fail with an explicit message naming the missing file.
2. Validate `tokens.paper.json` against `schemas/paperTokens.schema.json`.
3. Compute SHA-256 of each source file.
4. Copy files to `/tokens/raw/paper/` preserving names.
5. Write `_manifest.json`:
   ```json
   {
     "importedAt": "2026-04-23T20:15:00Z",
     "sources": [
       { "from": "tokens/source/paper/tokens.paper.json", "to": "tokens.paper.json", "sha256": "..." },
       ...
     ]
   }
   ```
6. **Seed authored files if missing.** `[D37]` Ensures exception-dependent stages have the files they need before `check-exceptions` or `normalize` runs:
   - `/tokens/authored/approvedPairs.json` + `approvedPairs.schema.json` — one or two example pairs plus `$description` field.
   - `/tokens/authored/sourceExceptions.json` + `sourceExceptions.schema.json` — empty `exceptions: []` array plus `$description` field.
   - If files already exist, they are untouched.
   - Seeded files are committed by the designer or design-system owner before first `normalize`.
7. Print a summary: `imported N files from <source> into /tokens/raw/paper/. <authored files seeded | authored files already present>.`

### 2.6 Failure modes

- Missing source directory or `tokens.paper.json` → fail, name the missing path.
- Destination exists and not `--force` → fail with "files already imported. Use --force to overwrite, or run `design-tokens normalize` to rebuild from existing raw."
- `tokens.paper.json` not valid JSON or not schema-valid → fail with schema errors.

### 2.7 Decisions implemented

`[D37]` Authored-file seeding happens at import, not normalize.

## 3. Command: `design-tokens normalize`

### 3.1 Purpose

Transform the raw paper snapshot into the normalized token structure defined by `[D1]`–`[D11]`. Paper source already uses canonical token names, so normalization is primarily file splitting, reference validation, density stubbing, and report generation.

The output is the source of truth everything downstream consumes.

### 3.2 Signature

```
design-tokens normalize [--dry-run] [--report <path>]
```

- `--dry-run` — compute normalization but don't write files; print diff summary.
- `--report` — write a detailed normalization report (every rename, every alias resolved, every warning) to the given path.

### 3.3 Expected inputs

`/tokens/raw/paper/tokens.paper.json` from the `import` stage.

### 3.4 Outputs

```
/tokens/normalized/
  primitives.colors.json
  primitives.numbers.json
  semantic.foundation.json      ← surface, content, stroke, shift
  semantic.feedback.json        ← alert, success, warning, info
  semantic.interactive.json     ← primary (from Emphasis)
  spacing.gap.json
  spacing.padding.json
  radius.json
  typography.roles.json         ← family/weight/tracking/case per role
  typography.sizes.json
  typography.line-heights.json
  components.button.json        ← passes through, flagged non-conformant
  themes/
    light.json                  ← theme mode definitions
  density/
    comfortable.json            ← stub, identical values [D17]
    snug.json                   ← stub, identical values [D17]
    compact.json                ← stub, identical values [D17]
  _normalization-report.json    ← renames, drifts, warnings
```

Note: `approvedPairs.json` is **not** a `normalize` output — it lives in `/tokens/authored/` and is hand-edited [D20, D34].

### 3.5 Step-by-step behavior

Each step cites the decisions it implements.

**Step 1 — Load and validate raw inputs.**
Load `/tokens/raw/paper/tokens.paper.json`. Fail loudly if schema is unexpected (missing required top-level families, missing `$type`, missing `$value`).

**Step 2 — Emit primitives.**

Read `primitives.colors` and `primitives.numbers` from the paper source and emit `primitives.colors.json` and `primitives.numbers.json` without renaming keys.

**Step 3 — Emit semantic families.** `[D2]`

Read `semantic.foundation`, `semantic.feedback`, and `semantic.interactive` from the paper source and emit the three normalized semantic files. Validate that semantic color references point to `primitives.*`.

**Step 4 — Emit spacing and radius.** `[D7, D8]`

Emit `spacing.gap.json`, `spacing.padding.json`, and `radius.json` from the paper source.

**Step 5 — Emit typography.** `[D9, D10, D11, D36, D38]`

Paper source already records canonical `typography.roles`, `typography.sizes`, and `typography.lineHeights`. Emit those as normalized files. If a future paper source records a paired-variant conflict plus an exception tiebreak, preserve `$extensions["leadbank.sourceExceptionApplied"]` on the emitted role.

**Step 6 — Stub density modes.** `[D13, D17]`

Emit `density/comfortable.json`, `density/snug.json`, `density/compact.json`, each containing the same values for `spacing.gap`, `spacing.padding`, and (future) component heights. In v1 the three files are byte-identical by design; phase 2 divergence requires a decision-doc update first.

Example of `sourceExceptionApplied` metadata on an emitted role:

```json
{
  "typography.roles.body-sm": {
    "$value": { "family": "Lead Sans Variable", "weight": "Regular", "tracking": 0, "case": "Original" },
    "$type": "typography",
    "$extensions": {
      "leadbank.sourceExceptionApplied": {
        "rule": "L7-weight-mismatch",
        "tiebreak": "leading",
        "reason": "temporary paper source conflict",
        "resolvedBy": "2026-Q2"
      }
    }
  }
}
```

**Step 7 — Pass-through components.** `[D27]`

Copy any `components.*` entries from paper source to normalized component files. Mark pre-anatomy component tokens with `$extensions.leadbank.nonConformant: true` until phase 2.

**Step 8 — Clarification on typography drifts and synthesis.** `[D33]`

The `normalize` command never synthesizes missing typography pairs or fills gaps in scales. Drifts, missing variants, and scale holes are flagged as warnings in the normalization report. Resolution is a design decision, not a tooling decision. (Authored-file seeding moved to `import` — see §2.5, `[D37]`.)

**Step 9 — Emit normalization report.**

`_normalization-report.json` contains:
- Every rename (old name → new name).
- Every alias resolved (token → target primitive).
- Every collapsed duplicate.
- Every typography drift warning.
- Every non-conformant marker.

This report is how designers understand what the CLI did to their data. Readable diff view is a phase 2 nice-to-have.

### 3.6 Failure modes (hard-stop)

- Orphan alias (alias points to primitive that doesn't exist).
- Primitive cleanup ambiguity (ghost key value diverges from canonical).
- Blue restructuring ambiguity (values can't be cleanly mapped to 100–1000).
- Paper source contains a top-level family or token shape not covered by `paperTokens.schema.json` — update the schema and decisions before normalizing.

### 3.7 Warnings (don't stop normalize, may fail lint)

- Collapsed scale warning (shift, surface).
- Typography drift warnings.
- Non-conformant component marker.

### 3.8 Decisions implemented

`[D1]` Tier structure. `[D2]` Three-family semantic split. `[D9–D11]` Typography normalization. `[D13, D17]` Density mode stubbing. `[D20]` `approvedPairs.json`. `[D27]` Components pass-through.

## 4. Command: `design-tokens lint`

### 4.1 Purpose

Validate normalized tokens against the rules in `DESIGN-DECISIONS.md`. Exit nonzero on violations so CI can block PRs.

### 4.2 Signature

```
design-tokens lint [--fix] [--json]
```

- `--fix` — apply auto-fixable corrections (rare; most violations require human review).
- `--json` — output in JSON for CI consumption.

### 4.3 Expected inputs

- `/tokens/normalized/*.json` from `normalize` stage.
- `/tokens/normalized/_normalization-report.json` — evidence for L7 validation (what conflicts were encountered, which exceptions were applied). `[D38]`
- `/tokens/authored/approvedPairs.json` — consumed by Rule L6.
- `/tokens/authored/approvedPairs.schema.json` — validates L6 input file structure.
- `/tokens/authored/sourceExceptions.json` — consumed by Rules L5 and L7. `[D36]`
- `/tokens/authored/sourceExceptions.schema.json` — validates exception entries.

### 4.4 Rules

**Rule L1 — Layer discipline.** `[D1]`

Walk every token. Resolve its `$value`. If it's a reference (string starting with `{`), verify the target is in a tier directly above the current tier. Cross-tier skips, sideways references, and downward references all fail.

Specifically allowed:
- `semantic.*` → `primitives.*` ✓
- `component.*` → `semantic.*` ✓
- `domain.*` → `semantic.*` ✓
- `component.*` → `primitives.*` ✗ (skipping semantic)
- `semantic.feedback.*` → `semantic.foundation.*` ✗ (sideways)

**Rule L2 — Mode completeness.** `[D3]`

Every token that exists in one theme mode must exist in every theme mode. Same for density. A token present in `light` but missing from `dark` (when dark exists) fails. A token present in `comfortable` but missing from `compact` fails.

In v1, this rule is trivial because `light` is the only theme mode and the three density modes are byte-identical. Rule exists to catch future regressions.

**Rule L3 — Gap/padding drift.** `[D7]`

For every key that exists in both `spacing.gap` and `spacing.padding`, check:
- If values match: pass silently.
- If values differ AND the token has a `$extensions.leadbank.justification` string: pass with a note ("documented drift").
- If values differ AND no justification: **warn** (not fail) with "gap.X (value) diverges from padding.X (value) without justification. Consider adding `$extensions.leadbank.justification` or aligning values."

Warnings do not block the build. This matches the rationale in `DESIGN-DECISIONS.md §4.1` — drift is a legitimate future need, not a bug.

**Rule L4 — Density affects only allowed tokens.** `[D14, D16]`

For every token that varies across density modes, verify it's in the allowed set:
- Allowed: `spacing.*`, component heights, `typography.line-heights.*`, `typography.sizes.*` (only for `body-sm`, `caption`, `code`, `label`).
- Disallowed: primitives, semantic colors, `radius`, `typography.roles.*`, `typography.sizes.*` for display/heading/title roles.

Violations fail.

**Rule L5 — Size floors.** `[D15, D33, D36]`

For each entry in `typography.sizes.*`, in each density mode, check against the floor for its classification:

- Roles rendering as readable text: minimum **12px**. `body-sm` specifically: minimum **13px**.
- Sizes marked `$extensions.leadbank.nonText: true`: exempt from text floor; validated under Rule L6 `non-text` contrast instead.

**Exception handling.** For each below-floor text size, consult the resolved exception set from `check-exceptions` (§1.5):

1. Exception matching `(tokenPath, "L5-size-floor")` found → **warn**. Include `reason` and `resolvedBy` in the lint report.
2. Exception is also `overdue: true` → **warn overdue** (still non-blocking). Message: `"exception resolvedBy date has passed; please confirm the paper source fix status."`
3. No exception → **fail**. Message: `"typography.sizes.X (Npx) below Mpx floor. Raise in paper source, or add a sourceExceptions entry with a target resolution date."`

Lint always lists active exceptions at the end of its output so technical debt stays visible.

**Rule L6 — Approved-pair contrast.** `[D19, D20, D21]`

For every entry in `/tokens/authored/approvedPairs.json`:

1. Validate entry against `approvedPairs.schema.json`. Fail if schema violation (missing `textSize`, `targetLevel`, invalid token reference).
2. Resolve `fg` and `bg` to hex values in the target theme mode.
3. Compute WCAG contrast ratio.
4. Compare to threshold derived from `textSize × targetLevel`:

   | textSize | targetLevel | threshold |
   |----------|-------------|-----------|
   | `normal` | `AA`        | 4.5:1     |
   | `large`  | `AA`        | 3:1       |
   | `non-text` | `AA`      | 3:1       |
   | `normal` | `AAA`       | 7:1       |
   | `large`  | `AAA`       | 4.5:1     |

   (WCAG AAA does not define a separate non-text level; `non-text × AAA` validates at the same 3:1 threshold and emits a note.)

5. Fail if ratio does not meet threshold.

**Encoded feedback-color rule** (`[D21]`): no `approvedPairs.json` entry may declare `fg` in the `semantic.feedback.*` namespace paired with a light `bg` (`surface.0` or `surface.1`) with `textSize: "normal"`. This is a hard-coded fail independent of the computed ratio, because even a feedback color that coincidentally passes 4.5:1 today could regress on a future value change and the design intent is that feedback colors are not body-text foregrounds.

**Rule L7 — Typography drift.** `[D11, D33, D36, D38]`

Note: by the time `lint` runs, paired-variant conflicts have already been resolved during `normalize` (Step 8) using the exception set. Rule L7 validates that resolution happened correctly using two evidence sources:

1. **Per-token metadata**: `$extensions["leadbank.sourceExceptionApplied"]` on emitted typography role tokens. Primary validation path — self-contained, survives independent lint runs on normalized-only checkouts.
2. **Normalization report**: `_normalization-report.json` conflict log. Supplementary evidence for cross-checking.

**Validation logic:**

For every `typography.roles.*` entry carrying `$extensions["leadbank.sourceExceptionApplied"]`:
- Verify the exception still exists in the current resolved exception set (from `check-exceptions`). Fail if it was removed from `sourceExceptions.json` but the normalized output still shows the metadata — means `normalize` has not been re-run after exception change. Message: `"stale exception metadata on typography.roles.X. Run design-tokens normalize."`
- Verify the `rule` and `tiebreak` on the token metadata match the current exception. Fail on mismatch.

For every conflict logged in `_normalization-report.json`:
- Verify a corresponding `sourceExceptionApplied` metadata exists on the emitted token. Fail if a conflict was reported but no metadata was emitted (means normalize's resolution path was broken).

**Hard-fail conditions** (should not occur if `normalize` succeeded; failsafe):
- A paired-variant conflict exists in normalized typography roles without a matching exception and tiebreak — fail with the token path and a pointer to add an exception.
- An L7 exception in the set has no `tiebreak` field — fail. (Also caught earlier by `check-exceptions`; this is defense in depth.)

**Warn-only drifts** (no emission blockage; resolution is a designer decision):
- Missing emphasis variants where the pattern implies one should exist (e.g., `Title-Md-Emphasis` paired with `Title-Md`).
- Missing `no-leading` pair for a style that exists in `leading` (or vice versa).
- Incomplete scale (e.g., code missing S, L, XL).
- **Cross-scale letter-spacing inconsistency**: different roles with different tracking values (e.g., `title-sm-emphasis` tracking −1% vs. `title-md-emphasis` tracking 0%). These are separate role values with no collapse conflict, so warn only.

**Paired-variant letter-spacing mismatch** (hard-fail during normalize, not here): when `leading/X` and `no-leading/X` of the same role+size disagree on tracking. Requires an `L7-tracking-mismatch` exception with `tiebreak`. Handled during normalize Step 8.

**Overdue exceptions** tagged from `check-exceptions` are reported here as `WARN overdue`, non-blocking.

**Rule L8 — Non-conformant components.** `[D27]`

For every entry in `components.*.json` marked `$extensions.leadbank.nonConformant: true`, emit a warning: "`components.X` is flagged non-conformant (phase 2 work). Current state accepted."

Warnings don't fail the build. This rule exists so the technical debt is visible in every lint run.

## 5. Command: `design-tokens build`

### 5.1 Purpose

Emit consumer-facing artifacts: CSS custom properties, TypeScript types, Tailwind config fragment, and the generated `DESIGN.md`. Stamps version headers on every output `[D26]`.

### 5.2 Signature

```
design-tokens build [--out <path>]
```

- `--out` — override default output directory (default: `/generated/`).

### 5.3 Expected inputs

`/tokens/normalized/*.json` from `normalize`, after `lint` passes.

### 5.4 Outputs

```
/generated/
  tokens.css               ← CSS custom properties, mode-cascaded
  token-types.d.ts         ← TypeScript types for autocomplete
  tailwind.tokens.ts       ← Tailwind theme extension
  token-reference.generated.md  ← full alphabetical token reference

/DESIGN.md                 ← the agent-facing current-state doc
CHANGELOG.md               ← appended on version bumps
```

### 5.5 Step-by-step behavior

**Step 1 — Verify lint passed.** If not, fail with "run `design-tokens lint` and fix violations first."

**Step 2 — Read version.** From `package.json` (or dedicated `tokens/version.json`). Embed in every output.

**Step 3 — Emit `tokens.css`.**

Structure:
```css
/* Lead Design Tokens v0.1.0 — generated from commit abc123f */

:root {
  /* primitives */
  --primitives-colors-blue-600: #...;
  ...
}

[data-theme="light"] {
  /* semantic */
  --semantic-foundation-surface-0: var(--primitives-colors-grey-50);
  ...
}

[data-density="snug"] {
  --spacing-gap-md: 16px;
  ...
}

[data-density="compact"] {
  --spacing-gap-md: 12px;
  ...
}

[data-density="comfortable"] {
  --spacing-gap-md: 20px;
  ...
}
```

In v1, the three density blocks emit identical values per `[D17]`; phase 2 populates differences.

**Step 4 — Emit `token-types.d.ts`.**

A union of all valid token paths for autocomplete in code:
```ts
export type TokenPath =
  | 'primitives.colors.blue.600'
  | 'semantic.foundation.surface.0'
  | ...;

export type ThemeMode = 'light';
export type DensityMode = 'comfortable' | 'snug' | 'compact';
```

**Step 5 — Emit `tailwind.tokens.ts`.**

Tailwind theme extension using the CSS custom properties:
```ts
export const leadTokens = {
  colors: {
    surface: {
      0: 'var(--semantic-foundation-surface-0)',
      1: 'var(--semantic-foundation-surface-1)',
    },
    ...
  },
  spacing: { ... },
  borderRadius: { ... },
  fontFamily: { ... },
  fontSize: { ... },
};
```

Imported in `tailwind.config.ts` via `theme.extend`.

**Step 6 — Emit `DESIGN.md`.** `[D32]`

See `DESIGN.md` for the template shape. Sections are:
- Version header
- Current modes (theme, density)
- Primitives (tables of colors + numbers)
- Semantic foundation / feedback / interactive (tables with references)
- Spacing (gap, padding)
- Radius
- Typography (roles, sizes, line-heights)
- Approved pairs (table)
- Component tokens (flagged non-conformant)
- Domain tokens (placeholder — phase 2+)
- Changelog pointer

**Step 7 — Emit `token-reference.generated.md`.**

Alphabetical dump of every token, its value, and its references. Developer-oriented rather than agent-oriented.

**Step 8 — Stamp all outputs with version header.** `[D26]`

### 5.6 Failure modes

- Lint has not passed (or not been run).
- Template parse error in DESIGN.md template.
- Write permission denied on output directory.

### 5.7 Decisions implemented

`[D25, D26, D32]`

## 6. CI integration

### 6.1 Pre-stage: decisions-log integrity `[D35]`

Before the normalize/lint/build chain runs, CI validates documentation integrity:

1. Parse `DESIGN-DECISIONS.md` decisions log (the `| D## | ... |` table).
2. Assert the log is monotonically numbered from D1 with no gaps.
3. **Parse all decision references** across `DESIGN-DECISIONS.md`, `cli-spec.md`, `DESIGN.md`, and any other markdown under `/docs/`. A "decision reference" is any bracketed expression matching the grammar (whitespace permitted around separators):

   ```
   reference  := '[' WS? id-or-range (WS? ',' WS? id-or-range)* WS? ']'
   id-or-range := 'D' digit+ (WS? ('–' | '-') WS? 'D'? digit+)?
   WS         := (' ' | '\t')+
   ```

   So all of these forms are references to parse:

   - Single: `[D1]`
   - Comma-separated list: `[D1, D3, D7]`
   - En-dash or hyphen range: `[D9–D11]`, `[D9-D11]`
   - Mixed: `[D1, D3–D5, D7]`
   - With or without whitespace: `[D1,D3]` and `[D1, D3]` both valid.

   A range `[Dn–Dm]` expands to all integers from n through m inclusive, each of which must resolve.

4. Fail if any parsed ID does not resolve to an entry in the log.
5. Fail if a decision is removed from the log but references to it still exist.

Implementation: ~50 lines of shell/Node. A regex like `/\[D\d+(?:\s*[,–-]\s*D?\d+)*\]/g` captures all reference expressions; a small parser expands ranges and commas. Worth the cost because broken cross-references silently decay the doc set.

### 6.2 Token pipeline

On any PR that touches `/tokens/**`:

1. Run `design-tokens normalize` — must succeed without hard failures.
2. Run `design-tokens lint` — must report zero failures. Warnings OK.
3. Run `design-tokens build`.
4. Diff-check: files in `/generated/` and `DESIGN.md` committed in the PR must byte-match the `build` output. If not, fail with "generated files are stale. Run `design-tokens build` and commit."

Additionally, on any PR that touches `/tokens/authored/`:

5. Validate `/tokens/authored/approvedPairs.json` against its co-located JSON Schema.
6. Run `design-tokens lint` — changes to approved pairs can newly fail L6 (contrast).

### 6.3 Release

On merge to main:

7. Inspect commit messages for Conventional Commits `feat`/`fix`/`BREAKING CHANGE`. Bump version accordingly `[D25]`.
8. Update `CHANGELOG.md`.
9. Tag the release.
