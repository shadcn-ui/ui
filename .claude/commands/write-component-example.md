# Write a Component Example (original shadcn convention)

Force UI follows shadcn/ui's original example convention: **many small, single-purpose
demo files**, each rendered bare and referenced one-per-section from the component's MDX
doc via `<ComponentPreview name="вА¶" />`.

> [!IMPORTANT]
> There are two example systems in this repo. This skill is about the **original shadcn
> one** in `apps/v4/examples/{radix,base}/`. Do **NOT** confuse it with the gallery files
> in `apps/v4/registry/bases/{radix,base}/examples/*-example.tsx` that use
> `Example` / `ExampleWrapper` / `<Example title="вА¶">`. Those are a separate showcase
> system. Demo files written here have **no titles, no cards, no wrapper chrome** вАФ the
> preview frame supplies the background.

## Where files live (React: radix + base)

| File | Purpose |
|---|---|
| `apps/v4/examples/radix/<component>-<feature>.tsx` | The demo (Radix base). |
| `apps/v4/examples/base/<component>-<feature>.tsx` | Identical demo for the Base UI base. **Always add both.** |
| `apps/v4/content/docs/components/radix/<component>.mdx` | The doc that references the demo by name. |

The filename (kebab-case, no extension) **is** the `name` used in MDX вАФ e.g.
`input-basic.tsx` вЖТ `name="input-basic"`. The registry index (`apps/v4/examples/__index__.tsx`)
is **autogenerated** by `scripts/build-registry.mts`; never hand-edit it.

## Keep every framework matching

A component's examples must be **parallel across all frameworks**: the same set of demos,
the same kebab-case `name`, the same realistic content and layout вАФ only the syntax and
import paths change. If you add `input-field` for React, add the Vue / Svelte / Ember
counterparts too (and reference them from each framework's MDX doc). Coverage should not
diverge between frameworks.

| Framework | Demo file location & naming | Component import | Icons | MDX reference |
|---|---|---|---|---|
| React (radix) | `apps/v4/examples/radix/<comp>-<feature>.tsx` | `@/styles/radix-force-ui/ui/<comp>` | `lucide-react` | `<ComponentPreview styleName="radix-force-ui" name="<comp>-<feature>" />` |
| React (base) | `apps/v4/examples/base/<comp>-<feature>.tsx` | `@/styles/base-force-ui/ui/<comp>` | `lucide-react` | `<ComponentPreview styleName="base-force-ui" name="<comp>-<feature>" />` |
| Vue | `packages/registry-vue/examples/<comp>/<PascalName>.vue` (grouped subdir, **PascalCase** file) вАФ `<script setup lang="ts">` | `@/ui/<comp>` | `lucide-vue-next` | `<ComponentPreview framework="vue" name="<comp>-<feature>" />` |
| Svelte | `packages/registry-svelte/examples/<comp>-<feature>.svelte` (flat, kebab) вАФ `<script lang="ts">` | `$lib/registry/ui/<comp>/index.js` | `@lucide/svelte/icons/<name>` | `<ComponentPreview framework="svelte" name="<comp>-<feature>" />` |
| Ember | `packages/registry-ember/examples/<comp>-<feature>.gts` (flat, kebab) вАФ `<template>` tag | `@/ui/<comp>` | `~icons/lucide/<name>` | `<ComponentPreview framework="ember" name="<comp>-<feature>" />` |

### Registration

| Framework | How it's registered |
|---|---|
| React (radix/base) | **Auto.** `pnpm registry:build` scans `apps/v4/examples/{radix,base}/` and regenerates `apps/v4/examples/__index__.tsx`. Drop the file, rebuild вАФ no manual entry. Never hand-edit `__index__.tsx`. |
| Vue | **Manual.** Add an entry to `packages/registry-vue/examples/_registry.ts` вАФ grouped (one `*-example` entry per component, listing each `.vue` file). |
| Svelte | **Manual.** Add an entry to `packages/registry-svelte/examples/_registry.ts` вАФ flat (one entry per file). |
| Ember | **Manual.** Add an entry to `packages/registry-ember/examples/_registry.ts` вАФ flat (one entry per file). |

### How `ComponentPreview` resolves a name

- **React (radix/base)**: resolves from `apps/v4/examples/__index__.tsx` first, then falls
  back to the gallery registry (`registry/bases/__index__.tsx`). This fallback is a trap: if
  you point a doc at a gallery name like `name="badge-example"`, it renders the chrome-heavy
  `Example`/`ExampleWrapper` showcase instead of a bare demo вАФ and the build still passes.
  Always point docs at the kebab name of an `apps/v4/examples/` demo.
- **Vue / Svelte / Ember**: the iframe renders a **separate, hand-maintained preview file**,
  NOT the registry example file. You MUST create it:

  | Framework | Preview file (what the docs render) | Import inside it |
  |---|---|---|
  | Vue | `apps/preview-vue/src/vue/<comp>-<feature>.vue` | `@/ui/<comp>` |
  | Svelte | `apps/preview-svelte/src/svelte/<comp>-<feature>.svelte` | `@/svelte-ui/<comp>/index.js` |
  | Ember | `apps/preview-ember/src/ember/<comp>-<feature>.gts` | `@/ember-ui/<comp>` |

  These are kebab-named, flat, and auto-discovered via `import.meta.glob` (no router edit
  needed). Note the preview import path **differs** from the registry example import path in
  the table above (e.g. Svelte preview uses `@/svelte-ui/вА¶`, registry uses
  `$lib/registry/ui/вА¶`). The registry example file + `_registry.ts` entry is for CLI
  distribution; the **preview file is what makes the docs render**. Create both.

After any change, run `pnpm registry:build` then `pnpm --filter=v4 build` (the latter builds
the three preview servers with Vite вАФ that step is what catches a broken/missing preview file
or an unknown CSS utility; `registry:build` alone does not).

Framework notes:
- The **`name` is always kebab-case** in every framework's MDX, even though Vue stores files
  as PascalCase in per-component subdirectories. Vue's registry (`packages/registry-vue/registry.ts`)
  groups one entry per component with multiple files; Svelte and Ember register one entry per file.
- Props translate to each framework's idiom: React/Vue/Svelte use `variant="вА¶"` / `class="вА¶"`;
  Ember uses `@variant="вА¶"` / `@class="вА¶"`.
- The MDX skeleton (sections, prose, `previewClassName`) is identical across frameworks; only
  the `framework=` / `styleName=` attribute, CLI prefix (`@force-ui-vue/вА¶`, `@force-ui-svelte/вА¶`),
  and code-block language differ. See `/import-shadcn-examples` for the full per-framework
  MDX transform tables.

## The core rule: one file = one concept

A demo shows **exactly one idea**, with a single instance of the component. Realistic,
contextual content вАФ never lorem/placeholder filler.

Two deliberate exceptions:
- **`<component>-demo.tsx`** вАФ the hero demo at the top of the doc. A small, representative
  composition (often 1вАУ3 elements), `export default`.
- **`<component>-variants.tsx`** вАФ the *one* place that enumerates the full variant axis
  side by side.

A contrast demo may show two instances when the point is the contrast (e.g.
`badge-icon.tsx` shows icon-left and icon-right). That's still one concept.

### Before writing a `-variants` / status / color demo: confirm the variant exists end-to-end

A demo that uses `variant="success"` is worthless if the variant isn't wired up. A variant
must exist in **all** of these or the demo renders wrong (or the build fails):

1. **CSS class** in `apps/v4/registry/styles/style-force-ui.css` вАФ
   `.cn-<comp>-variant-<name> { @apply вА¶ }`. (This file is Force UI's; safe to edit freely.)
2. **Component cva map** in *every* framework that ships the component вАФ React
   (`registry/bases/{radix,base}/ui/<comp>.tsx`), Vue (`packages/registry-vue/ui/<comp>/`),
   Svelte (`packages/registry-svelte/ui/<comp>/`), Ember (`packages/registry-ember/ui/<comp>/`).
   These drift independently; a variant added to React is often missing in the ports.
3. **Semantic color token** (if the variant uses one like `bg-success`/`text-on-success`):
   the `--color-*` mapping must exist in the `@theme inline` block of `apps/v4/app/globals.css`
   **and** in each `apps/preview-{vue,svelte,ember}/src/styles.css`. If a preview app lacks
   it, its Vite build dies with `Cannot apply unknown utility class bg-success`. Keep them in
   sync by running `apps/v4/scripts/sync-theme-css.mts` (it mirrors globals' `@theme inline`
   block into the previews); the raw `:root`/`.dark` palette also comes from that script.

Enumerate a `-variants` demo as a bare stack (`<div className="flex w-full flex-col gap-4">`
or `flex flex-wrap gap-2`) with the placeholder/label naming the variant
(`placeholder="Outline"`). Do **not** wrap each in `Field`/`FieldGroup` just to stack them.

### Counting examples
- Single-state/feature demos: **1** instance.
- `-variants`: **all** values of that prop.
- `-demo` (hero): a small representative composition.
- Never dump every variant into a feature demo, and never wrap a single control in
  `FieldGroup` just to stack variants вАФ that's the gallery anti-pattern.

## Anatomy of a demo file

```tsx
import {
  Field,
  FieldDescription,
  FieldLabel,
} from "@/styles/radix-force-ui/ui/field"
import { Input } from "@/styles/radix-force-ui/ui/input"

export function InputField() {
  return (
    <Field>
      <FieldLabel htmlFor="input-field-username">Username</FieldLabel>
      <Input
        id="input-field-username"
        type="text"
        placeholder="Enter your username"
      />
      <FieldDescription>
        Choose a unique username for your account.
      </FieldDescription>
    </Field>
  )
}
```

Conventions:
- **Imports**: UI from `@/styles/radix-force-ui/ui/<component>` (Base files use
  `@/styles/base-force-ui/ui/<component>`). Icons imported directly from `lucide-react`
  (or `@tabler/icons-react`), with `data-icon="inline-start"` / `data-icon="inline-end"`.
- **Export**: hero `-demo` files use `export default function <Component>Demo()`. Feature
  demos use a `export function <DescriptiveName>()`.
- **No chrome**: no `Example`, no `ExampleWrapper`, no `title` prop, no `bg-*`/card classes.
  Layout only when showing multiple items: a bare `<div className="flex flex-wrap gap-2">`.
- **Labeled controls**: use a single `Field` + `FieldLabel` + `FieldDescription`. Pair
  `id` with `htmlFor`. Reserve `FieldGroup` for genuine multi-field forms
  (`<component>-form.tsx`), not for stacking variants.
- **States**: disabled вЖТ `data-disabled` on `Field` + `disabled` on the control; invalid вЖТ
  `data-invalid` on `Field` + `aria-invalid` on the control.
- **Placeholders are realistic** (`"sk-..."`, `"Enter your username"`, `"Email"`), never
  the literal `"Placeholder"`.

### RTL demos (`<component>-rtl.tsx`)

```tsx
"use client"

import {
  useTranslation,
  type Translations,
} from "@/components/language-selector"
import { Field, FieldLabel } from "@/styles/radix-force-ui/ui-rtl/field"
import { Input } from "@/styles/radix-force-ui/ui-rtl/input"
```

Import from `ui-rtl/` (not `ui/`), define a `Translations` map with `en` / `ar` / `he`,
call `useTranslation`, and pass `dir` to the components.

## The MDX side вАФ what goes in the doc, not the demo

All prose, headings, install/usage snippets, and API tables live in MDX. The `.tsx` file
contains only the rendered demo. Standard section order:

```mdx
---
title: Input
description: A text input component for forms and user data entry.
base: radix
component: true
---

<ComponentPreview styleName="radix-force-ui" name="input-demo" previewClassName="*:max-w-xs" />

MDX rules:
- Each `## Examples` entry is `### Title` + an **optional one-sentence** description of the
  prop/API being shown + a single `<ComponentPreview>`.
- One `<ComponentPreview>` per `###`. The `name` must match a demo filename.
- Sizing/framing is controlled in MDX via `previewClassName` (`*:max-w-xs`, `p-6`,
  `h-[32rem]`, вА¶) вАФ **not** inside the demo.
- Always `styleName="radix-force-ui"` (or `base-force-ui`). Never an upstream demo style.

## Workflow

1. Create `apps/v4/examples/radix/<component>-<feature>.tsx` (single concept, bare).
2. Create the matching `apps/v4/examples/base/<component>-<feature>.tsx` (same demo, base
   import paths).
   (React demos register automatically on rebuild вАФ no manual entry.)
3. Create the matching demo for **every other framework** that already ships this component.
   For each framework this is **two files**: (a) the **preview file** that the docs iframe
   renders вАФ `apps/preview-{vue,svelte,ember}/src/{vue,svelte,ember}/<comp>-<feature>.{vue,svelte,gts}`
   вАФ and (b) the registry example file + a manual `examples/_registry.ts` entry (Vue:
   `examples/<comp>/<PascalName>.vue`; Svelte/Ember: flat kebab). Same content, same kebab
   `name`; mind the differing preview vs registry import paths.
4. Add a `### Section` + `<ComponentPreview name="<component>-<feature>" вА¶>` to **each**
   framework's MDX doc (`content/docs/components/{radix,vue,svelte,ember}/<component>.mdx`).
5. Rebuild the index + verify: `pnpm registry:build` then `pnpm --filter=v4 build`.
6. Confirm every `ComponentPreview name=` in every framework's MDX resolves to a registered
   demo (`examples/__index__.tsx` for React; each framework's `_registry.ts` + preview server
   otherwise).

## Anti-patterns (do not do)

- вЭМ Using `Example` / `ExampleWrapper` / `<Example title="вА¶">` in `apps/v4/examples/`.
- вЭМ Wrapping a single variant set in `FieldGroup` with `placeholder="Placeholder"`
  (that's the gallery style, the opposite of this convention).
- вЭМ Cramming multiple unrelated states into one demo file.
- вЭМ Putting prose/headings inside a `.tsx` demo, or styling/sizing inside the demo instead
  of `previewClassName`.
- вЭМ Adding only the radix file and forgetting the base counterpart.
- вЭМ Letting framework coverage diverge вАФ a demo in React but missing in Vue/Svelte/Ember,
  or the same concept built with different content/data across frameworks.
- вЭМ Using a non-kebab `name` in MDX (Vue files are PascalCase on disk, but the `name` is
  still kebab-case).
- вЭМ Adding a Vue/Svelte/Ember registry example file but forgetting the **preview-server
  file** (`apps/preview-{fw}/src/{fw}/<name>.{ext}`) вАФ that's the file the docs actually
  render; without it `ComponentPreview` shows "not found".
- вЭМ Pointing a React doc at a gallery `*-example` name вАФ it silently renders the
  `Example`/`ExampleWrapper` chrome via the registry fallback instead of a bare demo.
- вЭМ Shipping a `variant="вА¶"` demo without confirming the variant exists in the CSS, in the
  cva map of *all* frameworks, and (for color variants) in the preview apps' `@theme inline`.
- вЭМ Hand-editing the autogenerated `apps/v4/examples/__index__.tsx` (React is auto-registered
  via `pnpm registry:build`).
